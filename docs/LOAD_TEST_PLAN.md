## 1. 부하 테스트 배경 및 목적

### 1.1 배경

콘서트 예약 서비스는 **인기 공연 티켓 오픈 시 단시간에 대량의 트래픽이 집중**되는 특성을 가지고 있습니다.  
특히 다음과 같은 상황에서 시스템 장애가 발생할 가능성이 높습니다:

- 🎫 **티켓 오픈 시점**: 수천~수만 명의 동시 접속으로 대기열 토큰 발급 폭주
- 💺 **좌석 예약 경쟁**: 동일 좌석에 대한 동시 예약 요청 (동시성 이슈)
- 💰 **결제 처리**: 임시 예약 만료 전 대량 결제 요청 집중
- 🔄 **포인트 충전/차감**: 동일 사용자에 대한 동시 포인트 변경 요청

### 1.2 목적

| 목적 | 상세 내용 |
|------|----------|
| **시스템 한계 파악** | 현재 인프라에서 수용 가능한 최대 TPS 및 동시 사용자 수 확인 |
| **병목 지점 식별** | DB Connection, Lock 경합, Slow Query 등 병목 요소 사전 탐지 |
| **동시성 검증** | 낙관적 락/분산 락 기반 동시성 제어의 실효성 검증 |
| **적정 배포 스펙 도출** | CPU/Memory 자원 조절을 통한 최적 배포 스펙 산출 |
| **장애 예방** | 실 서비스 배포 전 장애 유발 시나리오를 사전 테스트하여 예방 |

## 2. 시스템 구성

### 2.1 아키텍처 개요

```
┌──────────┐     ┌──────────────────┐     ┌────────┐
│  Client  │────▶│  Spring Boot App │────▶│ MySQL  │
│  (k6)   │     │   (Java 21)      │     │  8.0   │
└──────────┘     │                  │────▶│        │
                 │  - Queue         │     └────────┘
                 │  - Concert       │     ┌────────┐
                 │  - Reservation   │────▶│ Redis  │
                 │  - Payment       │     │  7.2   │
                 │  - Point         │     └────────┘
                 └──────────────────┘     ┌────────┐
                                     ────▶│ Kafka  │
                                          └────────┘
```

### 2.2 기술 스택

| 구성 요소 | 기술 | 비고 |
|-----------|------|------|
| Application | Spring Boot 3.4.1 + Java 21 | HikariCP 커넥션 풀 (max: 3) |
| Database | MySQL 8.0 | 낙관적 락 (Seat Version) |
| Cache | Redis 7.2 | 대기열 관리, 분산 락 (Redisson) |
| Message Queue | Kafka (3 브로커) | 결제 완료 이벤트 발행 |
| 부하 테스트 도구 | k6 | JavaScript 기반 경량 도구 |

### 2.3 주요 API 목록

| API | Method | Endpoint | 인증 | 설명 |
|-----|--------|----------|------|------|
| 대기열 토큰 발급 | POST | `/queue/token` | ❌ | 대기열 진입 |
| 대기열 상태 조회 | GET | `/queue/status` | ✅ Token | 대기 순서 확인 |
| 예약 가능 날짜 조회 | GET | `/concerts/available-dates` | ✅ Token | 공연 일정 조회 |
| 좌석 목록 조회 | GET | `/concerts/seats?date=` | ✅ Token | 좌석 상태 조회 |
| 좌석 임시 예약 | POST | `/reservations` | ✅ Token | 5분간 임시 예약 |
| 결제 | POST | `/payment` | ✅ Token | 포인트 차감 + 확정 |
| 포인트 충전 | POST | `/points/charge` | ❌ | 포인트 충전 |
| 포인트 잔액 조회 | GET | `/points/balance?userId=` | ❌ | 잔액 확인 |
| 콘서트 랭킹 조회 | GET | `/concerts/ranking` | ❌ | 매진 랭킹 |

## 3. 테스트 대상 선정

### 3.1 테스트 대상 및 선정 이유

#### 🔴 **우선순위 1 (Critical)** - 동시성 이슈 및 데이터 정합성

| 대상 API | 선정 이유 |
|----------|----------|
| **좌석 임시 예약** (`POST /reservations`) | 동일 좌석에 대한 동시 예약 요청 시 **낙관적 락**으로 정합성 보장 필요. 데드락 가능성 점검 |
| **결제 처리** (`POST /payment`) | 포인트 차감 + 예약 확정 + Kafka 이벤트 발행이 **하나의 트랜잭션**으로 묶임. 가장 무거운 API |
| **포인트 충전** (`POST /points/charge`) | 동일 사용자의 동시 충전 요청 시 **잔액 정합성** 보장 필요 |

#### 🟡 **우선순위 2 (High)** - 대량 트래픽 처리

| 대상 API | 선정 이유 |
|----------|----------|
| **대기열 토큰 발급** (`POST /queue/token`) | 티켓 오픈 시 **가장 먼저 폭주**하는 API. Redis 부하 집중 |
| **대기열 상태 조회** (`GET /queue/status`) | 대기 중 사용자들의 **지속적 폴링** 요청. 높은 RPS 예상 |

#### 🟢 **우선순위 3 (Medium)** - 조회 성능

| 대상 API | 선정 이유 |
|----------|----------|
| **좌석 목록 조회** (`GET /concerts/seats`) | DB 조회 쿼리 성능 확인. **Slow Query** 가능성 점검 |
| **예약 가능 날짜 조회** (`GET /concerts/available-dates`) | 캐싱 여부에 따른 성능 차이 확인 |

## 4. 테스트 시나리오

### 4.1 시나리오 1: 대기열 토큰 발급 폭주 (Peak Test)

> **상황**: 인기 콘서트 티켓 오픈 시각에 수천 명이 동시 접속

| 항목 | 값 |
|------|-----|
| 목표 VUs (동시 사용자) | 500 → 1,000 → 2,000 |
| 테스트 시간 | 60초 |
| 예상 TPS | 1,000+ |
| 주요 관찰 지표 | 응답시간 p95, 에러율, Redis 연결 수 |

### 4.2 시나리오 2: 좌석 예약 경쟁 (Stress Test)

> **상황**: 50개 좌석에 대해 500명이 동시 예약 시도

| 항목 | 값 |
|------|-----|
| 목표 VUs | 100 → 300 → 500 |
| 테스트 시간 | 120초 (점진적 증가) |
| 예상 TPS | 300+ |
| 주요 관찰 지표 | 동시성 에러율, DB Lock 대기시간, Version 충돌 빈도 |

### 4.3 시나리오 3: 결제 처리 부하 (Load Test)

> **상황**: 임시 예약된 좌석에 대한 대량 결제 요청

| 항목 | 값 |
|------|-----|
| 목표 VUs | 50 → 100 → 200 |
| 테스트 시간 | 180초 |
| 예상 TPS | 100+ |
| 주요 관찰 지표 | 트랜잭션 처리시간, DB Connection Pool 사용률, Kafka 발행 지연 |

### 4.4 시나리오 4: 포인트 동시 충전 (Load Test)

> **상황**: 동일 사용자에 대한 동시 포인트 충전 요청

| 항목 | 값 |
|------|-----|
| 목표 VUs | 50 → 100 |
| 테스트 시간 | 60초 |
| 예상 TPS | 200+ |
| 주요 관찰 지표 | 잔액 정합성, 낙관적 락 충돌율 |

### 4.5 시나리오 5: 전체 사용자 플로우 (End-to-End Load Test)

> **상황**: 대기열 진입 → 좌석 조회 → 예약 → 결제 전체 플로우

| 항목 | 값 |
|------|-----|
| 목표 VUs | 100 → 200 |
| 테스트 시간 | 300초 |
| 시나리오 | 토큰 발급 → 상태 조회 → 날짜 조회 → 좌석 조회 → 예약 → 결제 |
| 주요 관찰 지표 | 전체 플로우 완료율, 평균 전체 소요시간 |

## 9. 예상 리스크 및 대응 방안

### 9.1 예상 병목 지점

| 병목 지점 | 원인 | 영향도 | 대응 방안 |
|-----------|------|--------|----------|
| **DB Connection Pool 고갈** | HikariCP max=3으로 매우 적음 | 🔴 Critical | Pool 사이즈 조절 테스트 |
| **좌석 예약 Lock 경합** | 동일 좌석 동시 예약 시 낙관적 락 충돌 | 🟡 High | 재시도 로직 확인, 실패율 측정 |
| **결제 트랜잭션 지연** | 포인트 차감 + 예약 확정 + Kafka 발행 | 🟡 High | 트랜잭션 분리 검토 |
| **Redis 병목** | 대기열 관리 요청 집중 | 🟢 Medium | Redis 메모리/연결 수 모니터링 |

### 9.2 데드락 가능성

- **좌석 예약 + 포인트 차감**: 서로 다른 테이블에 Lock 순서가 다른 경우
- **대응**: 부하 테스트 중 MySQL DeadLock 모니터링 (`SHOW ENGINE INNODB STATUS`)
